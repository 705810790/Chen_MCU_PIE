# code-onboard

`code-onboard` 是一个面向代码可读性和接手维护的 Codex Skill，用于阅读并理解已有工程，为项目自有源码添加高价值注释，并更新工程内部 `README.md` 作为快速上手文档。

它不负责新增功能，也不负责阶段 commit/tag。它的目标是让一个已经成型的代码工程更容易被后来者读懂、调试和维护。

## 背景

很多项目在功能完成后会出现：

- 代码能跑，但模块职责不够清晰
- 状态机、中断、任务、队列、协议映射等关键逻辑缺少解释
- README 不能反映当前工程结构和使用方法
- 新接手的人需要重新追源码才能知道数据怎么流动
- 注释太少或注释只是重复代码，没有说明意图

`code-onboard` 用来在代码完成后做一次“接手友好化”整理。

## 适用场景

- 项目已有可用代码基线
- 想给自有源码补充文件头、函数注释、结构体/枚举/宏注释
- 想解释状态机、中断、任务、队列、硬件约束或协议映射
- 想把工程内部 README 改成快速上手文档
- 想排除 SDK、第三方库、自动生成代码和编译产物，只处理项目自有代码

## 使用方式

推荐提示：

```text
使用 $code-onboard，阅读当前工程代码，给项目自有源码添加注释，并更新工程 README.md。
```

建议明确输入：

- 真实工程目录
- 工程内部 README 路径
- 需要注释的源码范围
- 需要排除的目录
- 可选项目框架或阶段说明路径

示例：

```text
使用 $code-onboard，只处理 User/App 和 User/main.c，更新 FreeRTOS_Core/README.md，跳过 FreeRTOS、SDK、obj 和自动生成文件。
```

## 注释范围

适合注释：

- 项目自有 `.c/.h/.cpp/.hpp` 文件
- 模块入口、公共 API、任务、ISR、回调、协议处理函数
- 状态机、队列、事件、硬件访问函数
- 结构体、枚举、宏分组、配置常量
- 映射表、码表、寄存器表
- 不容易从代码直接看懂的关键逻辑段

不建议处理：

- 第三方库
- SDK / vendor 代码
- 自动生成代码
- 编译产物
- 与当前阅读和接手无关的历史文件

## README 内容

`code-onboard` 会把工程内部 `README.md` 整理成快速上手文档，通常包含：

- 项目概述
- 当前代码基线
- 核心能力
- 硬件引脚或运行环境
- 目录结构
- 核心模块调用关系
- RTOS 任务 / 队列 / 事件说明
- 构建与调试
- 版本历史

README 应描述当前状态，不写冗长开发日志，不替代外部阶段说明文档。

## AI 推荐用法

完整整理：

```text
使用 $code-onboard，先读项目框架和 README，再阅读源码，给项目自有代码补充注释，并更新 README。
```

只处理部分模块：

```text
使用 $code-onboard，只注释 src/protocol 和 src/app，跳过 drivers/vendor。
```

只更新 README：

```text
使用 $code-onboard，阅读代码后只更新 README，不修改源码注释。
```

## 注意事项

- 注释应解释意图、数据流、约束和关键决策，不要机械重复代码
- 不要在注释阶段改业务逻辑
- 如果发现代码问题，应先报告，除非用户明确要求修复
- README 面向接手开发者，不写详细阶段验收日志
- 如果 README 版本和 Git tag 不一致，应在结果中说明
