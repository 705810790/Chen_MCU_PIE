# project-frame

`project-frame` 是一个面向 AI 协作开发的 Codex Skill，用于在正式写代码前读取需求文件夹、主需求文档和参考资料文件夹，提炼项目目标、约束、参考资料、模块边界、调试方式和阶段计划，生成一份可持续更新的项目框架 Markdown。

它的目标不是直接写代码，而是先把项目的“路线图”和“后续 AI 必须知道的信息”整理清楚，AI知晓项目大方向，让后续开发可以按框架逐步推进。

## 背景

在 AI 参与项目开发时，如果没有统一框架，后续很容易出现：

- 中大型项目需要多轮对话才可完成，统一好框架后，可随时续接之前的工作进度，避免每次任务重复前提
- 需求和参考资料没有被完整吸收
- AI 不清楚哪些资料是权威参考，哪些只是示例
- 模块拆分、阶段计划和验收标准不明确
- 调试工具、人工硬件操作、用户特别提醒没有固定记录，每次都需要重新提醒
- 后续阶段开发时每次都要重新解释项目背景


`project-frame` 用来解决这个问题：它会先把需求和资料整理成动态框架，后续项目推进时再按实际情况更新框架。

## 适用场景

- 新项目开始前，需要先梳理需求和技术路线
- 用户已经准备了需求文件夹和参考资料文件夹
- 项目还没有创建工程，或刚创建工程但还没有清晰架构
- 希望 AI 生成模块拆分、阶段计划、验收标准和调试规则
- 希望后续 AI 能读取框架后继续推进项目，而不是每次重新解释

## 使用方式

推荐提示：

```text
使用 $project-frame，读取我的需求文件夹、主需求文档和参考资料文件夹，生成项目框架 md。
```

`project-frame` 会主动询问：

- 需求文件夹路径
- 主需求文档路径
- 参考资料文件夹路径
- 真实工程目录或预计工程目录
- 框架 Markdown 输出路径
- 调试方式、工具链、用户特别提醒

## 框架内容

默认生成的项目框架包含：

- 用户提醒和执行注意事项
- 项目目标
- 需求来源
- 参考资料和复用策略
- 已知硬件、软件、工具链、目录约束
- 总体流程
- 推荐工程结构
- 模块拆分
- 初始化 / 执行 / 调用顺序
- 调试和验证方法
- 阶段实现计划
- 文档和版本规则
- 待确认问题
- 实施原则

## AI 推荐用法

新项目开始时：

```text
使用 $project-frame，需求文件夹是 <path>，参考资料文件夹是 <path>，框架输出到 <path>。
```

如果已有框架但需要更新：

```text
使用 $project-frame，根据当前实现情况更新项目框架。
```

如果参考资料很多：

```text
使用 $project-frame，先列出参考资料候选文件，不要盲目读取整个文件夹。
```

## 注意事项

- 不要让 AI 在没有框架前直接写大量代码
- 参考资料文件夹应先筛选候选文件，再读取与项目目标相关的内容
- 框架是动态文档，阶段完成后应和真实工程状态对齐
- 框架不需要每次临时修改都做 Git 版本管理
